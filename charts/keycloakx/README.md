## movetokube/postgres-operator integration

Set `postgresOperator.enabled: true` to render `Postgres` and `PostgresUser`
custom resources alongside Keycloak. The operator (deployed separately as
`ext-postgres-operator`) reconciles these CRs against the configured RDS
instance and writes the per-user credentials into a Kubernetes Secret named
`<secretName>-<role>` in this chart's release namespace. Reference that
Secret from `keycloakx.extraEnv` via `valueFrom.secretKeyRef` to feed
Keycloak's `KC_DB_*` environment variables.

```yaml
postgresOperator:
  enabled: true
  databases:
    keycloak:
      dropOnDelete: false
      users:
        keycloak:
          privileges: OWNER
          secretName: keycloak
```

### Pinning to a specific operator instance

When multiple `ext-postgres-operator` deployments coexist (e.g. one per RDS), set `POSTGRES_INSTANCE` on the operator and a matching `postgres-operator/instance` annotation on the CRs the keycloakx chart emits. The annotation can be set globally for all CRs, or scoped per-database / per-user:

```yaml
postgresOperator:
  enabled: true
  # Global default — applies to every Postgres + PostgresUser CR.
  annotations:
    postgres-operator/instance: "main"
  databases:
    keycloak:
      # Optional: override the Postgres CR's annotations for this database.
      # annotations:
      #   postgres-operator/instance: "reporting"
      users:
        keycloak:
          privileges: OWNER
          secretName: keycloak
          # Optional: override the PostgresUser CR's annotations for this user.
          # annotations:
          #   postgres-operator/instance: "reporting"
```

Replacement semantics — the most specific annotations map wins entirely, no merging.
